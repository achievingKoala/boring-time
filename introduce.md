# 📄 产品需求文档 (PRD): SilentStats Web

**目标：** 记录用户手机处于"无声状态"的总时长（即用户远离抖音、B站等干扰应用的时间）。

**核心理念：** 放弃"实时监控"，采用"端点追溯"和"焦点争夺"逻辑。

---

## 1. 核心技术逻辑

### 1.1 锁屏计时补全 (Back-filling Time)

**痛点：** 网页锁屏后 JavaScript 脚本通常会停止运行。

**对策：** 记录挑战开始时间戳。当用户重新回到页面时，用当前时间减去开始时间，得到本次总时长。

### 1.2 干扰应用探测 (Audio Focus Conflict)

**痛点：** 网页无法知道用户在后台开了什么 App。

**对策：** 利用浏览器音频焦点机制。

- 网页开始时播放一段循环音频（`p1.mp3`），占住音频焦点。
- 只要用户开启抖音、B站、网易云音乐等有声 App，系统会强制抢占焦点并暂停网页音频。
- 网页监听 `onpause` 事件，记录打断开始时间，状态显示"检测到干扰"。
- 打断后**不主动抢回**，等用户回到页面后手动点开始继续。

---

## 2. 功能清单

### 2.1 任务执行逻辑

**【开始挑战】**
- 用户点击按钮触发音频播放（浏览器要求用户交互才能播放）。
- 记录 `sessionStartTime` 时间戳，新建一条时间轴记录。
- 若是续接（被打断后回来点开始），保留原有计时，不重置。

**【后台运行】**
- 用户锁屏或切后台，音频持续运行。
- 若用户打开抖音等 App，音频焦点被抢占，触发 `onpause`，记录打断开始时间，状态显示"检测到干扰"，发送系统通知（需 HTTPS + 用户授权）。

**【回到页面】**
- 若期间被打断：结束当前时间轴记录，显示"已暂停"，等用户手动点开始继续。
- 若未被打断（正常锁屏）：自动恢复播放，继续计时。

**【结束专注】**
- 用户手动点击"结束专注"按钮，结算本次数据。
- 显示：本次总时长 / 被打断时长 / 纯净专注时长。

### 2.2 今日专注计数

- 顶部实时显示今日累计纯净专注时长（精确到秒）。
- 只有音频播放且未被打断时才累加。
- 以日期为 key 存入 localStorage，每天自动清零。

### 2.3 今日时间轴

- 24 小时横条，绿色块表示每次专注时段。
- 正在进行的时段闪烁显示。
- 条目列表显示每段的开始→结束时间和时长。
- 每次只有一个"进行中"状态，新开始前自动关闭未结束的记录。
- 数据存入 localStorage，刷新不丢失，第二天自动清零。

### 2.4 系统通知

- 被打断时发送通知"检测到干扰"。
- 需要 HTTPS 环境 + 用户授权（iOS Safari 不支持）。

---

## 3. 技术栈

- **前端：** 原生 HTML / CSS / JavaScript
- **样式：** Tailwind CSS（CDN 引入）
- **音频：** 本地 `p1.mp3`，循环播放
- **存储：** localStorage（无数据库，无后端）
- **部署：** Vercel / GitHub Pages（静态托管，需 HTTPS 以支持通知）

---

## 4. 判定逻辑矩阵

| 状态组合 | 系统行为 | 结果判断 |
|---|---|---|
| 网页可见 + 音频播放 | 正常运行 | 计入纯净时间 |
| 网页隐藏 + 音频播放 | 用户锁屏（自律） | 计入纯净时间（回来后自动恢复） |
| 网页隐藏 + 音频暂停 | 用户刷抖音/听歌 | 计入打断时间，不计入纯净时间，回来需手动点开始 |
