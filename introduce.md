# 📄 产品需求文档 (PRD): SilentStats Web

**目标：** 记录用户手机处于"无声状态"的总时长（即用户远离抖音、B站等干扰应用的时间）。

**核心理念：** 放弃"实时监控"，采用"端点追溯"和"焦点争夺"逻辑。

---

## 1. 核心技术逻辑

### 1.1 锁屏计时补全 (Back-filling Time)

**痛点：** 网页锁屏后 JavaScript 脚本通常会停止运行。

**对策：** 记录挑战开始时间戳。当用户重新回到页面时，用当前时间减去开始时间，得到本次总时长。

### 1.2 干扰应用探测 (Audio Focus Conflict)

**痛点：** 网页无法知道用户在后台开了什么 App。

**对策：** 利用浏览器音频焦点机制。

- 网页开始时播放一段循环音频（静音或测试用有声音），占住音频焦点。
- 只要用户开启抖音、B站、网易云音乐等有声 App，系统会强制抢占焦点并暂停网页音频。
- 网页监听 `onpause` 事件，记录打断开始时间；监听 `onplay` 事件，记录打断结束时间，累加打断时长。

---

## 2. 功能清单

### 2.1 任务执行逻辑

**【开始挑战】**
- 用户点击按钮触发音频播放（浏览器要求用户交互才能播放）。
- 记录 `sessionStartTime` 时间戳。

**【后台运行】**
- 用户锁屏或切后台，音频持续运行。
- 若用户打开抖音等 App，音频焦点被抢占，触发 `onpause`，记录打断开始时间，状态显示"检测到干扰"。
- 若音频焦点恢复，触发 `onplay`，累加本次打断时长，状态恢复"静默探测中"。

**【回归自动结算】**
- 用户回到网页，触发 `visibilitychange`，自动结算，无需手动点击。
- 计算：总时长 = 当前时间 - 开始时间
- 计算：纯净时长 = 总时长 - 累计打断时长
- 只将纯净时长计入历史总计。

### 2.2 结算展示

回到页面后显示三项数据：
- **本次总时长**：从开始到回来的完整时间
- **被打断时长**：期间音频焦点被抢占的累计时间
- **纯净专注时长**：总时长减去打断时长，计入历史

### 2.3 极简 UI

- 主计时器：显示本次已运行时长
- 状态标签：显示"静默探测中"或"检测到干扰"
- 顶部历史总计：累计纯净专注分钟数（localStorage 持久化）

---

## 3. 技术栈

- **前端：** 原生 HTML / CSS / JavaScript
- **样式：** Tailwind CSS（CDN 引入）
- **存储：** localStorage（无数据库，无后端）
- **部署：** Vercel / GitHub Pages（静态托管）

---

## 4. 判定逻辑矩阵

| 状态组合 | 系统行为 | 结果判断 |
|---|---|---|
| 网页可见 + 音频播放 | 正常运行 | 计入纯净时间 |
| 网页隐藏 + 音频播放 | 用户锁屏（自律） | 计入纯净时间（回来后补算） |
| 网页隐藏 + 音频暂停 | 用户刷抖音/听歌 | 计入打断时间，不计入纯净时间 |
