既然我们要遵循 levelsio 的开发哲学，这份需求文档必须极其精简 (Lean)。他最讨厌繁琐的流程，所以我们直接聚焦于：如何用最少的代码，解决“网页在锁屏状态下统计无声时间”这个痛点。

以下是这份去掉了所有冗余功能、纯粹为了解决核心技术问题的需求文档。

📄 产品需求文档 (PRD): SilentStats Web

目标： 记录用户手机处于“无声状态”的总时长（即用户远离抖音、B站等干扰应用的时间）。

核心理念： 放弃“实时监控”，采用“端点追溯”和“焦点争夺”逻辑。

1. 核心技术逻辑 (Technical Logic)

1.1 锁屏计时补全 (Back-filling Time)

痛点：网页锁屏后 JavaScript 脚本通常会停止运行。

对策：使用 localStorage 记录开始时间戳。当用户重新回到页面时，对比“当前时间”和“上次活跃时间”，将中间的缺口（Gap）补算进总时长。

1.2 干扰应用探测 (Audio Focus Conflict)

痛点：网页无法知道用户在后台开了什么 App。

对策：利用浏览器音频焦点机制。

网页开始时播放一段 1 秒的 无声 Base64 音频 并设置循环。

只要用户开启抖音、B站、网易云音乐等有声 App，系统会强制抢占焦点并暂停网页音频。

网页监听 onpause 事件。一旦音频在后台被中断，立即记录一个“违规时刻”。

2. 功能清单 (Functional Features)

2.1 任务执行逻辑

【开始挑战】：

向用户申请一次点击交互（以开启音频播放权限）。

写入当前时间戳 startTime 到本地存储。

启动无声音频循环。

【静默运行】：

用户锁屏或切后台，网页音频持续运行。

若此时用户打开抖音，系统掐断网页音频，JS 即使在后台也会触发最后一次 onpause 记录：violationDetected = true。

【回归结算】：

用户回到网页（触发 visibilitychange）。

检查 violationDetected 标记。

若无违规：计算 Now - lastActiveTime，累加到“有效无声时长”。

若有违规：提示“检测到声音干扰”，违规期间的时间不予计入。

2.2 极简 UI 界面

主计时器：显示当前已累计的“手机无声总时间”。

实时状态：显示“静默探测中”或“检测到干扰”。

今日汇总：记录用户今天通过这种方式远离屏幕的总分钟数。

3. 技术栈 (Stack)

前端：原生 HTML / CSS / JavaScript (不再引入任何库，甚至不选 React/Vue)。

样式：直接通过 CDN 引入 Tailwind CSS 快速排版。

存储：浏览器 localStorage（无数据库，无后端，0 维护成本）。

服务器：Vercel / GitHub Pages (静态托管)。

4. 判定逻辑矩阵 (Matrix)

状态组合系统行为结果判断网页可见 + 音频播放正常运行计入无声时间网页隐藏 + 音频播放用户锁屏（真自律）计入无声时间（回来后续加）网页隐藏 + 音频暂停用户刷抖音/听歌（作弊）不计入时长，显示挑战失败5. 部署与传播 (Levelsio 策略)

快速发布 (MVP)：只写一个 index.html 文件，今天下午就上线。

冷启动：录一段“我刷抖音被网页抓到了”的短视频发到小红书或 X，利用噱头获客。

变现预留：若用户量增加，可增加“自律挑战金”功能（用户押注 1 块钱，失败了钱归你）。

6. 后续迭代 (Not in MVP)

V2：增加环境光探测（必须把手机扣在桌面上才开始计费）。

V3：支持多端同步（虽然 Levels 会建议你先只做一个移动端）。

专家点评：

这份文档去掉了所有浮夸的功能。它不再关心“用户到底在干嘛”，它只关心一件事：我的静音音频有没有被挤断？ 只要没断，就是“无声时间”。这非常符合 Levelsio 的风格——找到最省力的技术杠杆，解决最大的用户痛点。
